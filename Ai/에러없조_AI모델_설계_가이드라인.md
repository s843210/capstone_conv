## 1. 우리 데이터가 어떤 데이터인가

### 1-1. 기본 현황

| 항목 | 수치 |
|------|------|
| 총 행 수 | 713,756행 |
| 기간 | 2024-03-01 ~ 2026-03-31 (약 25개월, 730일) |
| 고유 상품 수 | 4,245개 |
| 일별 평균 행 수 | 약 978행 |
| sales = 0 비율 | **71.9%** |
| sales > 0 비율 | 28.1% (200,749행) |
| sales < 0 (반품) | 3건 (무시 가능) |

### 1-2. 판매량(sales) 분포 특성

- **대부분이 0이다:** 전체의 71.9%가 판매 없음. "대부분의 상품이 대부분의 날에 안 팔린다."
- **팔려도 소량이다:** 양수 판매의 중위값은 1개, 평균 2.36개, 최대 52개.
- **가장 많은 판매량은 1개:** sales=1이 10만6천 행으로 양수 중 53%를 차지.
- **오른쪽 꼬리가 길다:** 가끔 20~52개까지 팔리는 날이 있지만 극소수.

### 1-3. 왜 이게 "특수한" 문제인가

이런 데이터를 수요예측 분야에서 **간헐적 수요(Intermittent Demand)**라고 부른다. 특징은 다음과 같다:

1. **Zero-Inflation:** 0이 전체의 70% 이상을 차지하기 때문에, 일반적인 회귀 모델은 "일단 0이라고 예측"하는 쪽으로 쏠린다. MSE를 최소화하면 0 예측만으로도 loss가 낮아지기 때문이다.
2. **Long Tail:** 양수 영역에서도 1~3개가 대부분이고 가끔 수십 개가 팔린다. 평균(MSE) 기준 최적화를 하면 큰 값에 끌려가고, 중위값(MAE) 기준이면 전부 1로 예측하게 된다.
3. **상품 수명 차이:** 4,245개 상품 중 2년 내내 팔리는 상품도 있고, 1일만 등장하는 상품도 있다.

이 세 가지 특성 때문에 단순한 단일 모델(Random Forest 등)만으로는 구조적 한계가 있다.

## 2. 모델 구조: Two-Stage LightGBM

### 2-1. 핵심 아이디어

하나의 질문("내일 이 상품이 몇 개 팔릴까?")을 두 개로 분리한다:

```
Q1. "내일 이 상품이 팔리기는 할까?" → Stage 1 (분류 모델)
Q2. "팔린다면 몇 개 팔릴까?"       → Stage 2 (회귀 모델)

최종 예측 = Q1의 확률 × Q2의 예측량
```

이 구조를 수요예측/통계학에서 **Hurdle Model**이라고 부른다. "첫 번째 장벽(hurdle)을 넘느냐 마느냐" → "넘었다면 얼마나 가느냐"로 나누는 것이다.

### 2-2. 왜 이 구조가 수학적으로 올바른가

기대값의 분해(Law of Total Expectation)라는 확률론 기본 공식에서 나온다:

```
E[sales] = P(sales=0) × 0 + P(sales>0) × E[sales | sales>0]
         = P(sales>0) × E[sales | sales>0]
```

즉, **전체 기대 판매량 = 팔릴 확률 × 팔릴 때의 기대량**이다. 우리가 하려는 것은 이 공식의 두 항을 각각 별도의 모델로 추정하는 것이며, 통계적으로 완전히 정당한 접근이다.

### 2-3. 왜 Random Forest 단일 모델 대신 이 구조를 쓰는가

| 관점 | Random Forest 단일 모델 | Two-Stage LightGBM |
|------|------------------------|---------------------|
| Zero 처리 | 0이 71.9%인 데이터에서 리프 노드 평균이 0 쪽으로 쏠림 | Stage 1이 "팔릴지 안 팔릴지"를 명시적으로 분리 |
| 학습 방식 | Bagging (각 트리가 독립적으로 학습 후 평균) | Boosting (이전 트리의 실수를 다음 트리가 보정) |
| 손실함수 | MSE 고정 | Tweedie 손실 등 유연한 선택 가능 |
| 범주형 피처 | One-Hot Encoding 필요 | LightGBM이 내부적으로 최적 분할 |
| 폐기 최소화 | "2.3개 팔릴 것"만 출력 | "판매 확률 15%" → 발주 안 함 결정 가능 |
| 대시보드 활용 | 단일 숫자 | 확률 × 수량의 2축으로 상품 분류 가능 |

### 2-4. LightGBM이란?

**Gradient Boosted Decision Trees(GBDT)**의 한 종류이다.

- **Random Forest:** 여러 트리를 "동시에" 독립적으로 만들고 투표 (Bagging)
- **LightGBM:** 첫 번째 트리가 틀린 부분(잔차)을 두 번째 트리가 학습하고, 두 번째가 틀린 부분을 세 번째가 학습하는 것을 반복 (Boosting)

비유하면 RF는 "10명이 각자 시험 보고 다수결", LightGBM은 "한 명이 채점 후 다음 사람이 틀린 문제만 다시 풀고, 또 다음 사람이 고치고..." 를 반복하는 것이다.

LightGBM이 XGBoost보다 빠른 이유는 **Leaf-wise** 성장(가장 loss가 큰 리프를 먼저 분할)을 쓰기 때문이다. XGBoost는 Level-wise(같은 깊이 리프를 모두 분할)라서 같은 분할 횟수 대비 LightGBM이 더 빠르게 loss를 줄인다.

## 3. Stage 1: 분류 모델 상세

### 3-1. 목표

> "내일 이 상품이 1개라도 팔릴까?"

### 3-2. Target 정의

```python
df['sold'] = (df['sales'] > 0).astype(int)
# 결과: 0(안 팔림)이 71.9%, 1(팔림)이 28.1%
```

### 3-3. 모델

**LightGBM Classifier** (`objective='binary'`)

### 3-4. 불균형 처리

0이 71.9%로 불균형하기 때문에, 아무것도 안 하면 모델이 "무조건 0"이라고 예측해도 정확도 71.9%가 나온다. 쓸모없는 모델이 "높은 정확도"로 보이는 함정이다.

**해결:** `is_unbalance=True` 설정 → LightGBM이 양성(판매 발생)에 높은 가중치를 부여하여 "실제로 팔리는 날을 맞추는 것"에 집중하게 됨.

### 3-5. 우리 데이터에서 Stage 1이 잘 작동할 수밖에 없는 근거

실제 데이터 분석 결과, "팔릴지 안 팔릴지"를 결정하는 요인이 매우 뚜렷하다:

| 요인 | 판매율 (높음) | 판매율 (낮음) | 배율 |
|------|:---:|:---:|:---:|
| 요일 | 화요일 39.8% | 일요일 6.4% | 6.2배 |
| 공휴일 (평일 중) | 비공휴일 37.5% | 공휴일 5.4% | 6.9배 |
| 학사일정 | 정규수업 34.2% | 비수업 14.3% | 2.4배 |
| 최근 판매 이력 | rolling_7_mean>0 → 37.7% | rolling_7_mean=0 → 12.8% | 2.9배 |
| 안전재고 | safety_stock>0 → 69.6% | safety_stock=0 → 23.0% | 3.0배 |

요일 + 공휴일 + 학사일정 3개만으로도 "팔릴지 안 팔릴지"의 큰 그림을 상당히 잘 나눌 수 있다.

### 3-6. 평가 지표

| 지표 | 의미 | 우리 프로젝트에서의 해석 |
|------|------|------------------------|
| **Precision** | "팔린다고 예측한 것 중 실제 팔린 비율" | 낮으면 → 불필요한 발주 → **폐기** 발생 |
| **Recall** | "실제 팔린 날 중 맞춘 비율" | 낮으면 → 필요한 상품 미발주 → **품절** |
| **F1 Score** | Precision과 Recall의 조화 평균 | 종합 성능 |

> **폐기 최소화가 목표**이므로 Precision을 살짝 더 중시하는 방향이 맞다. 즉, 애매하면 "안 팔린다"고 예측하는 게 폐기를 줄이는 방향이다.

## 4. Stage 2: 회귀 모델 상세

### 4-1. 목표

> "팔린다면 몇 개 팔릴까?"

### 4-2. 학습 데이터

**sales > 0인 행만 사용** (약 200,749행). 0인 행은 Stage 1이 처리하므로 Stage 2는 "팔리는 상황에서의 수량"에만 집중한다.

### 4-3. 모델: 가중평균 앙상블

```
Stage 2 예측 = 0.6 × LightGBM Regressor + 0.4 × XGBoost Regressor
```

**왜 두 모델을 합치는가:** LightGBM은 Leaf-wise 성장으로 빠르게 loss를 줄이지만 과적합 경향이 있고, XGBoost는 Level-wise로 좀 더 안정적이다. 두 모델의 실수 패턴이 다르기 때문에 평균을 내면 개별 모델보다 안정적인 예측이 나온다. 이를 Wisdom of Crowds 효과라 하며, 수학적으로 bias-variance decomposition에서 variance가 줄어드는 것에 해당한다.

> 가중치 0.6:0.4는 초기값이며, 검증 데이터에서 최적 비율을 탐색해야 한다.

### 4-4. 손실함수: Tweedie

```python
# LightGBM
objective='tweedie', tweedie_variance_power=1.5

# XGBoost  
objective='reg:tweedie', tweedie_variance_power=1.5
```

**왜 Tweedie인가:**

일반 MSE는 큰 값에 끌려간다. 우리 데이터에서 대부분 1~3개인데 가끔 30~52개 팔리는 날이 있으면, MSE는 그 큰 값을 맞추려고 전체적으로 예측을 높이게 된다. 결과적으로 "1개 팔릴 걸 3개로 예측" → 폐기 발생.

Tweedie 분포는 0에서 확률 질량을 갖고 양수 영역에서는 연속인 특수한 분포로, 분산이 평균의 거듭제곱에 비례(`Var ∝ μ^p`)한다. 이 특성 때문에 평균이 작은 영역(1~3개)에서는 보수적으로, 평균이 큰 영역(20~30개)에서는 유연하게 예측한다. 우리 sales 분포와 정확히 일치하는 특성이다.

### 4-5. 핵심 피처 (상관계수 기반, sales>0)

| 피처 | 상관계수 | 의미 |
|------|:---:|------|
| `rolling_7_mean` | 0.65 | **가장 강력.** 최근 7일 판매 평균 |
| `rolling_7_std` | 0.65 | 판매 변동성. 큰 변동 = 예측 어려운 상품 |
| `lag_7` | 0.61 | 7일 전 판매량 |
| `safety_stock` | 0.58 | 안전재고. 꾸준히 팔리는 상품의 신호 |
| `lag_1` | 0.51 | 어제 판매량 |
| `lag_3` | 0.31 | 3일 전 판매량 |

### 4-6. 평가 지표

| 지표 | 의미 |
|------|------|
| **MAE** | 평균 절대 오차. "평균적으로 몇 개 틀리는가" |
| **WAPE** | 가중 절대 백분율 오차. 판매량이 큰 상품의 오차에 더 가중치 |
| **Bias** | 양수면 과대예측(폐기↑), 음수면 과소예측(품절↑) |

## 5. 최종 예측 결합

```python
# Stage 1: 판매 확률
p_sell = stage1_model.predict_proba(X_test)[:, 1]   # 0~1 사이 값

# Stage 2: 조건부 판매량 (앙상블)
qty_if_sell = 0.6 * lgbm_reg.predict(X_test) + 0.4 * xgb_reg.predict(X_test)

# 최종 예측
final_prediction = p_sell * qty_if_sell
```

### 예시

| 상품 | 조건 | Stage 1 (확률) | Stage 2 (수량) | 최종 예측 | 발주 판단 |
|------|------|:---:|:---:|:---:|------|
| 농심)육개장사발면 | 화요일, 정규수업, rolling_7_mean=5 | 0.85 | 4.2 | **3.57** | 발주 4개 |
| 마즈)트윅스 | 일요일, 방학, rolling_7_mean=0.3 | 0.04 | 2.1 | **0.08** | 발주 안 함 |
| 에너지음료 | 목요일, 시험기간, rolling_7_mean=8 | 0.72 | 9.3 | **6.7** | 발주 6~8개 |

**핵심:** P(sell)이 낮은 상품은 Stage 2가 아무리 높은 수치를 예측해도 최종값이 작아져서, **폐기를 구조적으로 줄이는 메커니즘**이 된다.

## 7. 데이터 분할

시계열 데이터이므로 **반드시 시간 순서대로 분할**한다. 일반적인 랜덤 분할(train_test_split)을 쓰면 "미래 데이터로 과거를 예측"하는 정보 누수(leakage)가 발생한다.

```
학습(Train): 2024-03-01 ~ 2025-09-30  (459,145행, 64%)
검증(Valid): 2025-10-01 ~ 2025-12-31  (140,165행, 20%)
테스트(Test): 2026-01-01 ~ 2026-03-31 (114,446행, 16%)
```

| 구간 | 포함 시즌 | Zero율 | 용도 |
|------|----------|:---:|------|
| 학습 | 1학기+여름방학+2학기 (1.5년) | 67.1% | 모델 학습 |
| 검증 | 2학기 후반 (10~12월) | 79.2% | 하이퍼파라미터 튜닝 |
| 테스트 | 겨울방학+1학기 초 (1~3월) | 82.2% | **최종 성능 보고** |

> 테스트셋 Zero율이 82.2%로 높은 것은 정상이다. 1~2월이 겨울방학이라 대부분 안 팔리는 시즌이며, "방학에도 정확히 예측하는가"를 검증하는 현실적 테스트가 된다.

## 11. 주의사항 및 확인 필요 사항

### 11-1. 데이터 관련

1. **이전 통합 파일(sales_2024_03_01_to_2026_03_31_features.xlsx)의 sales가 2배로 되어 있었음.** 새 데이터(`데이터변환.zip` 내 파일들)가 정확한 원본이며, 반드시 이 데이터를 사용해야 한다.
2. **음수 sales 3건** (반품/취소 추정) → 학습 시 제거 또는 0으로 치환.
3. **is_holiday가 주말을 포함**하고 있다. is_holiday=1인 날 중 토·일이 100% 포함되어 있으며, 평일 중 공휴일도 1로 표시된다. day_of_week와 정보 중복이 있으나, 모델이 자동으로 유용한 쪽을 선택하므로 둘 다 유지한다.

### 11-2. 모델 관련

1. **safety_stock은 leakage가 아니다** — 시간에 따라 동적으로 변하는 rolling 기반 값임을 확인했다. 안전하게 피처로 사용 가능.
2. **product_sell_rate 파생 시 반드시 학습 기간 데이터만으로 계산**해야 한다. 테스트 기간이 섞이면 leakage.
3. **가중치(0.6:0.4)는 초기 설정값**이며, 검증 데이터에서 그리드 서치로 최적화해야 한다.
